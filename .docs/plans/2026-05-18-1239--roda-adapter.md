# Better Auth Roda Adapter Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a first-class, public Roda integration for Better Auth Ruby as a complete monorepo package.

**Architecture:** Add a `better_auth-roda` gem that wraps the existing Rack-compatible Better Auth core with Roda plugin APIs, helpers, and SQL migration tasks. Keep shared authentication behavior in `packages/better_auth`; keep Roda-only assumptions inside the Roda package.

**Tech Stack:** Ruby 3.2+, Rack 3-compatible Better Auth core, Roda 3.x, RSpec, Rack::Test, StandardRB, pnpm for docs.

---

## Public API And Structure

- Add canonical gem package `packages/better_auth-roda` and alias package `packages/openauth-roda`.
- Public require paths:
  - `require "better_auth/roda"`
  - `require "openauth/roda"`
- Public constants/modules:
  - `BetterAuth::Roda`
  - `BetterAuth::Roda::Configuration`
  - `BetterAuth::Roda::MountedApp`
  - `BetterAuth::Roda::Migration`
- Roda usage target:

```ruby
class App < Roda
  plugin :better_auth

  better_auth at: "/api/auth" do |config|
    config.secret = ENV.fetch("BETTER_AUTH_SECRET")
    config.base_url = ENV["BETTER_AUTH_URL"]
    config.database = :memory
    config.email_and_password = {enabled: true}
  end

  route do |r|
    r.better_auth

    r.get "dashboard" do
      current_user.to_json
    end
  end
end
```

- Do not add a Sequel-specific default. Roda uses core adapters explicitly, with SQL migration tasks like Sinatra.

## Test Plan

- Roda package specs:
  - Default and custom mount paths dispatch `/ok`.
  - Trailing slash parity matches core behavior.
  - `SCRIPT_NAME`/`PATH_INFO` split and nested Rack mount cases work.
  - Plugin endpoints receive cookies and emit `Set-Cookie`.
  - Server-only endpoints remain unreachable.
  - Origin and callbackURL checks stay active.
  - Unexpected endpoint errors follow `on_api_error` callback/throw semantics.
  - Helpers resolve real cookie sessions, bearer sessions, stale cleanup cookies, and do not leak sessions between requests.
  - `require_authentication` returns plain 401 or JSON 401 based on `Accept`.
  - Roda subclass apps can configure their own Better Auth mount without inheriting duplicate-configuration state from a parent class.
  - Helper session lookup preserves application `Set-Cookie` headers when Better Auth appends cleanup cookies.
  - Public `require "better_auth/roda"` works in a clean Ruby process.
  - Migration render/generate/migrate tasks cover plugin schema, dialect env vars, unsupported adapters, SQL splitting, and failed statement behavior.
- Alias/workspace specs:
  - `openauth-roda` loads `BetterAuth::Roda`.
  - `.release.yml` version manifests include Roda packages.
  - Root alias package test includes `Roda`.
- Verification commands:

## Assumptions

- Scope is the complete public package path: `better_auth-roda` plus `openauth-roda`, docs, CI, and release wiring.
- Database surface stays framework-neutral: no Sequel default in v1.
- Roda dependency should target Roda 3.x, e.g. `>= 3.0`, `< 4`, based on current Roda docs and RubyGems.
- No release version bump beyond adding new Roda package version files aligned with the current repository release manifest.
